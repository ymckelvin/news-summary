Where it began: &#8220;Ralph Wiggum&#8221; loop Exactly a year ago, software engineer Geoffrey Huntley published the article &#8216; Ralph Wiggum as a software engineer &#8217;. The name references the naive son of the local police chief in The Simpsons, who is extremely eager to always be helpful. In engineering, &#8220;Ralph&#8221; is intended to continuously nudge the agent in the right direction. Geoff described it: &#8220;Ralph is a technique. In its purest form, Ralph is a Bash loop: while :; do cat PROMPT.md | claude-code ; done Ralph can replace the majority of outsourcing at most companies for greenfield projects. It has defects, but these are identifiable and resolvable through various styles of prompts. That&#8217;s the beauty of Ralph - the technique is deterministically bad in a nondeterministic world.&#8221; The article expands on the idea of the Ralph loop: Start the agent with a prompt that captures the task and defines a goal Create a plan of work, each item with a success criteria Start the loop: Take one item per loop When the agent is done, check if the goal is achieved If not: start the agent again, with a clear context window Start loops if needed: spawn subagents as and when necessary Geoff published the experiments he did with this approach, such as building a new programming language last summer, and said it requires skill: &#8220;Engineers are still needed. There is no way this is possible without senior expertise guiding Ralph. Anyone claiming that engineers are no longer required and a tool can do 100% of the work without an engineer is peddling horses***.&#8221; The &#8220;Ralph method&#8221; blew up late last year with the arrival of better models which were surprisingly capable of building ambitious projects. Software engineer Matt Pocock created a tutorial, &#8216; Ship working code while you sleep &#8217; with the Ralph Wiggum technique. He said: &#8220;One of the dreams of coding agents is that you can wake up in the morning to working code, where your coding agent has worked through your backlog. It has spit out a whole bunch of code for you to review, and it works.&#8221; Before the Ralph loop, Matt did this in two steps: Ask the agent to create a detailed plan for the work, with tasks broken out Then, in a sequential order, have the agent complete each subtask in a separate run Pre-Ralph: The agent completes one step at a time in a single context window. Source: Matt Pocock A problem with this approach is there&#8217;s no easy way to add new tasks to the &#8220;masterplan&#8221;. Software engineers know that most plans do need to be modified, often while the work is ongoing. In contrast, with Matt&#8217;s take on the Ralph method, the &#8220;masterplan&#8221; is continuously updated in a &#8220;master PRD&#8221;. Here&#8217;s the prompt he gives the agent: Choose the next feature: Find the highest-priority feature to work on and work only on that feature Have tests pass : check that the tests pass (via pnpm test) Update the master tracker : update the PRD with the work done Log work: append your progress to the progress.txt file Commit : make a git commit of the feature This style of working is more of a &#8220;dynamic Kanban&#8221;: &#8220;dynamic Kanban&#8221; style of working. Source: Matt Pocock The &#8220;Ralph method&#8221; is all about working around context window limitations. Back in mid-2025, the maximum size of a context window was around 200,000 tokens. That&#8217;s not enough for more ambitious tasks, so it&#8217;s necessary to break up agent runs into smaller ones and run them, one by one. In this context, here&#8217;s where the Ralph method works: Have a goal for a project, and keep running (or re-running) agents until this goal is reached Persist work done in a &#8220;compressed&#8221; manner on the filesystem (as logs or an updated plan) Start agents with fresh context to minimize &#8220;context rot&#8221; Allow each agent to add or modify the &#8220;masterplan&#8221; if needed 2. The /goal command ships in all major harnesses For a few months, building a Ralph loop meant doing it yourself: setting up the loop, state tracking, deciding how the agent can add tasks and when to stop. But things changed once coding harnesses made it easy to run these loops. April: Codex ships /Goals About six months after the &#8220;Ralph technique&#8221; started gaining wider traction, Codex shipped the &#8220;Goals&#8221; feature in Codex. From the documentation : &#8220;Goals are persistent objectives in Codex that keep a thread working toward a defined outcome across turns. A Goal gives Codex a completion condition: what should be true, how success should be checked, and what constraints must stay intact.&#8221; Also from the docs (emphasis mine): &#8220;A normal prompt says: do this next thing. A Goal says: keep working until this outcome is true. In a normal request, Codex works through the immediate instruction, reports a result, and waits. With a Goal, Codex has a durable target attached to the thread. After a turn finishes, it can inspect the current evidence and decide whether the objective is satisfied.
